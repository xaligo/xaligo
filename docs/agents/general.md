## 1. Creating a Diagram (Full Workflow)

### Step 1 — Find service IDs

```bash
# Search by keyword (case-insensitive)
grep -i "ec2"        etc/resources/aws/service-index.csv
grep -i "rds"        etc/resources/aws/service-index.csv
grep -i "lambda"     etc/resources/aws/service-index.csv
grep -i "cloudfront" etc/resources/aws/service-index.csv
```

`service-index.csv` format: `id,ServiceName`

Example output:
```
27,Amazon EC2
117,Amazon RDS
1178,Amazon CloudFront
13,AWS Lambda
```

Use the numeric ID (first column) in `.xal` files and `services.csv`.

### Step 2 — Create services.csv

Create a file listing every service icon to embed in the diagram legend.

**Format:** `id,OfficialName,Abbreviation,Summary,Usage,Notes`

Rules:
- Column 1 must be the numeric catalog ID from `service-index.csv`
- `Abbreviation` is used as the icon label (keep it short, ≤6 chars)
- Lines starting with `#` are comments and ignored
- Trailing commas for empty columns are fine

```csv
# My Architecture — service list
# id,OfficialName,Abbreviation,Summary,Usage,Notes
1179,Amazon Route 53,R53,DNS service,Domain resolution,
1182,Elastic Load Balancing,ELB,Load balancer,Traffic distribution,
27,Amazon EC2,EC2,Virtual server,Application tier,
117,Amazon RDS,RDS,Relational DB,Data persistence,
13,AWS Lambda,Lambda,Serverless,Event processing,
```

Save as e.g. `examples/my-arch-services.csv`.

### Step 3 — Create a .xal layout file

`.xal` is an XML-based DSL. The root element must be `<frame>`.

**Minimal template:**
```xml
<frame width="1200" height="820" class="pa-4">
  <aws-cloud title="AWS Cloud">
    <region title="us-east-1">
      <vpc title="VPC (10.0.0.0/16)">
        <availability-zone title="AZ a">
          <public-subnet title="Public Subnet">
            <item id="1182" />   <!-- ELB -->
          </public-subnet>
          <private-subnet title="Private Subnet">
            <item id="27" />     <!-- EC2 -->
            <item id="117" />    <!-- RDS -->
          </private-subnet>
        </availability-zone>
      </vpc>
    </region>
  </aws-cloud>

  <connection src="1182" dst="27" />
  <connection src="27" dst="117" />
</frame>
```

**Key rules:**
- `<item id="N" />` embeds one service icon; `N` = catalog ID
- `<item />` (no id) = spacer slot (no icon, but layout space reserved)
- `<connection src="A" dst="B" />` draws an elbow arrow; must be a **direct child of `<frame>`**, placed **after** all layout tags
- `row="N"` on a subnet/container = number of icon columns in the icon grid
- `item-size="N"` on `<frame>` sets icon size in px (native CLI default 32)

**Available AWS group containers:**

| Tag | Border Color |
|---|---|
| `<aws-cloud>` | black |
| `<aws-account>` | `#E7008A` |
| `<region>` | `#00A1C9` (dashed) |
| `<availability-zone>` | `#00A1C9` (dashed) |
| `<vpc>` | `#8C4FFF` |
| `<public-subnet>` | `#3F8624` |
| `<private-subnet>` | `#00A1C9` |
| `<security-group>` | `#CC0000` (dashed) |
| `<auto-scaling-group>` | `#E7601B` (dashed) |

Save as e.g. `examples/my-arch.xal`.

### Step 4 — Generate the Excalidraw file

```bash
mkdir -p output
.bin/xaligo generate excalidraw \
  --xal examples/my-arch.xal \
  -o output/my-arch.excalidraw \
  --services examples/my-arch-services.csv
```

`--services` is **required**. It adds a legend panel on the right side of the frame.

On success: no output, exit code 0, file created at `output/my-arch.excalidraw`.

**Verify the output file exists:**
```bash
ls -lh output/my-arch.excalidraw
```

## 2. Adding Service Icons to an Existing .excalidraw File

Use `add service` when you want to append icons to a file that already exists
(e.g., a file generated by `generate xal`).

```bash
# Single icon
.bin/xaligo add service \
  --name "Amazon EC2" \
  --file output/existing.excalidraw

# Bulk from CSV (legend placed on the RIGHT of the frame)
.bin/xaligo add service \
  --list examples/services.csv \
  --file output/existing.excalidraw
```

Legend placement:
- `--list` mode → legend stacked on the **right** side of the frame
- `--name` mode → legend stacked on the **left** side of the frame

## 3. Generating an AWS Infrastructure Hierarchy .xal

Use `generate xal` to auto-generate a `.xal` file for a standard AWS layout.

```bash
.bin/xaligo generate xal \
  --clouds 1 \
  --accounts 1 \
  --regions 2 \
  --azs 2 \
  --az-layout staggered \
  --subnets 2 \
  --spacing both \
  --start top \
  --paper A4 \
  --orientation landscape \
  -o output/infra.xal
```

All flags are **required**. Then convert to Excalidraw:

```bash
.bin/xaligo generate excalidraw \
  --xal output/infra.xal \
  -o output/infra.excalidraw \
  --services examples/services.csv
```

## 4. Modifying Go Code

### Directory of relevant files

| File | Responsibility |
|---|---|
| `pkg/controller/add.go` | `add service` command + `RunAddServiceBatch` |
| `pkg/controller/generate.go` | `generate xal` + `generate excalidraw` |
| `pkg/controller/render.go` | `render` command + `RunRender` |
| `internal/parser/parser.go` | `.xal` XML parser → AST |
| `internal/layout/layout.go` | Layout engine (stack, row, group) |
| `internal/excalidraw/scene.go` | Excalidraw JSON builder (`BuildJSON` / `BuildJSONWithFS`) |
| `internal/repository/icon.go` | SVG lookup, base64 encoding; also `LookupCatalogByIDFS`, `SvgToDataURLFS` for WASM |
| `internal/repository/builder.go` | `MakeText`, `MakeImage` element builders |
| `internal/repository/service_list.go` | `ReadServiceList` (file path) + `ReadServiceListFromReader` (io.Reader, for WASM) |
| `internal/config/config.go` | Config loading from `etc/resources/aws/app.yaml` |
| `etc/resources/aws/assets.go` | `//go:embed service-catalog.csv svg` — exposes `awsassets.Assets fs.FS` for WASM |
| `cmd/wasm/main.go` | WASM entry point; uses `awsassets.Assets` to expose JS globals |

### Coding conventions

- Error wrapping: `fmt.Errorf("context: %w", err)` — never `panic`
- Excalidraw elements: `map[string]interface{}` — do not define new structs for elements
- Package names: lowercase single words (`controller`, `repository`, `entity`)
- After editing: always run `go build ./...` and confirm zero output

### Calling config methods

```go
cfg := config.New()
cfg.AssetDir()              // path to etc/resources/aws/svg
cfg.ServiceCatalogCSVPath() // path to service-catalog.csv
cfg.OutputFramesDir()       // path to output/aws-frames
```

### Looking up a service icon by catalog ID

```go
ce, err := repository.LookupCatalogByID(cfg.ServiceCatalogCSVPath(), catalogID)
// ce.DataURL  — "data:image/svg+xml;base64,..."
// ce.Service  — "Amazon EC2"
// ce.SVGFile  — "Arch_Amazon-EC2_64.svg"
```

### Building Excalidraw elements

```go
imgEl := repository.MakeImage(id, x, y, w, h, fileID, bgColor, seed)
txtEl := repository.MakeText(id, x, y, w, h, text, fontSize, color, bold, seed)
```

### Reading / writing a scene

```go
scene, err := repository.ReadScene(filePath)
scene.Elements = append(scene.Elements, imgEl)
err = repository.WriteScene(scene, filePath)
```

## 6. Common Mistakes and How to Avoid Them

| Mistake | Correct approach |
|---|---|
| Forgetting `--services` on `generate excalidraw` | Always include `--services <csv>`; it is required |
| Using `<connection>` inside a group tag | `<connection>` must be a **direct child of `<frame>`**, after all layout elements |
| Wrong catalog ID | Always verify with `grep -i "<name>" etc/resources/aws/service-index.csv` |
| Output directory missing | Run `mkdir -p output` before generating |
| Calling `cfg.AssetPackageDir()` or `cfg.SvcCatalogCSV` directly | Use `cfg.AssetDir()` and `cfg.ServiceCatalogCSVPath()` (these are the exported methods) |
| `panic` in error handling | Return `error`; never use `panic` |
| Editing files with shell redirection (`echo > file`) | Use file creation tools; never use shell redirection to overwrite source files |

## 9. WASM Build and npm Packages

### Background

All AWS SVG icons and `service-catalog.csv` are embedded into the WASM binary via `//go:embed`
in `etc/resources/aws/assets.go`.
The WASM artifact is therefore fully self-contained and does not access the OS filesystem at runtime.

### Building the WASM artifact

```bash
make build-wasm
# Writes: packages/xaligo/wasm/xaligo.wasm
#         packages/xaligo/wasm/wasm_exec.js
```

Internally:
```bash
GOOS=js GOARCH=wasm go build -o packages/xaligo/wasm/xaligo.wasm ./cmd/wasm
cp "$(go env GOROOT)/lib/wasm/wasm_exec.js" packages/xaligo/wasm/wasm_exec.js
```

### Exposed JavaScript globals

After the WASM is loaded, two functions are available on the global scope:

| Global | Signature | Description |
|---|---|---|
| `xaligoRender` | `(xal: string) => string` | DSL → Excalidraw JSON |
| `xaligoRenderWithServices` | `(xal: string, servicesCsv: string) => string` | DSL + services CSV → Excalidraw JSON with legend |

### TypeScript wrapper (`@ryo-arima/xaligo`)

Entry point: `packages/xaligo/src/index.ts`

```typescript
import { loadXaligo } from "@ryo-arima/xaligo";

const xaligo = await loadXaligo();         // idempotent; loads WASM on first call
const json   = await xaligo.render(xal);
const json2  = await xaligo.renderWithServices(xal, servicesCsv);
```

`loadXaligo()` accepts an optional `wasmUrl` parameter to override the default path
to `xaligo.wasm` (useful in VS Code extension host environments).

### npm package layout

| Package | Directory | Description |
|---|---|---|
| `@ryo-arima/xaligo` | `packages/xaligo/` | WASM + TypeScript wrapper |

### Common mistakes (WASM)

| Mistake | Correct approach |
|---|---|
| Using `wasm_exec.js` from `misc/wasm/` | Use `$(go env GOROOT)/lib/wasm/wasm_exec.js` (path changed in Go 1.21+) |
| Accessing the OS filesystem inside WASM | Use `awsassets.Assets` (embed.FS); no OS filesystem calls at WASM runtime |
| Passing `nil` fsys when using WASM path | Pass `awsassets.Assets` via `BuildJSONWithFS`; native CLI path passes `nil` |
| Forgetting the Go build tag | `cmd/wasm/main.go` has `//go:build js`; it is excluded from native builds automatically |
